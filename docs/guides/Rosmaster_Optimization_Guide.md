# Rosmaster Optimization Guide: Hardware and Software

This document provides a comprehensive guide on optimizing the communication between the Jetson Orin NX and the Rosmaster Expansion Board. It covers hardware-level connection improvements, USB kernel tuning, and software-level algorithm optimizations.

---

## 1. Hardware Optimization: Switching from USB to Hardware UART

### Why Switch?
The standard USB connection relies on the Jetson's USB bus polling and an internal USB-to-TTL convertor chip (like CH340 or CP2102) on the interface board. By switching to the Jetson's native hardware UART bus, you completely bypass the USB stack, significantly reducing CPU overhead, polling intervals, and latency.

### Jetson Orin NX (40-Pin Header) Pinout
Hold the Orin NX such that the 40-pin header is on your right. You will use **UART 1** (`/dev/ttyTHS0` or `/dev/ttyTHS1`).

```text
               (Edge of Jetson Board)
            ---------------------------
       3.3V [ 1] [ 2] 5V
      I2C_1 [ 3] [ 4] 5V
      I2C_1 [ 5] [ 6] GND   <--------- Connect to Rosmaster GND
      GPIO4 [ 7] [ 8] TXD   <--------- Connect to Rosmaster RX
        GND [ 9] [10] RXD   <--------- Connect to Rosmaster TX
      UART2 [11] [12] I2S_2
      I2S_2 [13] [14] GND
```

### Rosmaster Expansion Board Header
Find the 4-pin white JST-XH receptor near the Microcontroller. 

```text
[ 5V ]  (DO NOT CONNECT)
[ GND]  <--------- Connect to Jetson Orin NX Pin 6 (GND)
[ TX ]  <--------- Connect to Jetson Orin NX Pin 10 (RXD)
[ RX ]  <--------- Connect to Jetson Orin NX Pin 8 (TXD)
```

**Wiring Rules:**
1. Cross connect `TX` to `RX` and `RX` to `TX`.
2. Connect the `GND` pins together. 
3. **DO NOT** connect the `5V` pins. Powering one board from the other via this pin can damage the logic levels or the board.

## 2. USB Port Optimization: Tuning the Latency Timer

If changing physical wires is not an option and you must stick to the USB cable, the biggest bottleneck is the Linux Kernel's USB latency timer. 

### The Problem
By default, the Linux FTDI/USB-Serial drivers set a `latency_timer` of **16 milliseconds**. This means the kernel is artificially waiting up to 16ms to batch bytes together before turning them over to your Python process. For high-frequency IMU reads or PID control, 16ms is fatal.

### The Solution
You can drop this polling latency down to **1ms** for near real-time serial speeds.

To check your current latency:
```bash
cat /sys/bus/usb-serial/devices/ttyUSB0/latency_timer
```

To set the latency to 1ms (Requires `sudo`):
```bash
sudo sh -c 'echo 1 > /sys/bus/usb-serial/devices/ttyUSB0/latency_timer'
```

*Note: You can permanently configure this using a `udev` rule so that it persists across reboots.*

## 3. Software Optimizations: Refactoring `Rosmaster_Lib.py`

The default `Rosmaster_Lib.py` is written as a monolithic script that generates a lot of Python overhead. Below are the primary bottlenecks and how to refactor them.

### A. Buffer-Based Serial Reading (Fixing the 1-Byte bottleneck)
**Current Issue:** The `__receive_data` loop blocks the thread while calling `self.ser.read()` **one byte at a time** continuously. The context switching generated by requesting millions of single bytes from the OS taxes the CPU massively.
**Optimization:** Use `in_waiting` to read large chunks of bytes in a single OS call, pushing them into a `bytearray` buffer. Search the buffer for the `0xFF 0xFC` frame headers, validate the checksum, and then pop the processed packets off the front.

### B. Precompiled Struct Operations
**Current Issue:** The `__parse_data` loop dynamically slices byte arrays like `bytearray(ext_data[0:2])` and calls `struct.unpack('h', ...)` hundreds of times a second.
**Optimization:** Move struct definitions to the module scope level:
```python
import struct
_unpack_3_shorts = struct.Struct('<hhh').unpack_from
```
Using `_unpack_3_shorts(ext_data, 6)` against the memory buffer directly is significantly faster than slicing multiple new Python objects each frame.

### C. Thread Safety
**Current Issue:** A background thread is writing to internal variables (`self.__gx`, `self.__gy`, `self.__gz`) while your main ROS nodes might be reading them. Because IMU and Odometry data are composite states (X, Y, Z coordinates), the lack of thread-locking can cause torn reads (e.g., getting a new X reading but an old Y reading).
**Optimization:** Add a `threading.Lock()` wrapped around block setter and getter methods, ensuring absolute thread-safety when fetching coordinate frames.

### D. Eliminating Hardcoded Blocking Delays
**Current Issue:** Native write commands intentionally block the execution thread via `time.sleep(.002)` after every transmission. This artificially limits maximum throughput and can block the main ROS application loop. 
**Optimization:** Write into a thread-safe asynchronous queue, leaving the delays to a dedicated dispatch thread. This allows your main loop to fire commands at unlimited frequency without stalling.

---
By applying these combined hardware and software improvements, the Jetson Orin NX will achieve microsecond-latency control loops with minimal CPU load.
